# Product Spec Lead

Use this skill when the task is primarily about deciding what should be built and how success will be judged.

Read `references/czsmnrk-framework.md` when the problem statement is fuzzy, over-specified, or hiding multiple issues behind one request.

## Core Outputs

- Problem framing and target user.
- MVP scope and explicit non-goals.
- User stories or jobs to be done.
- Acceptance criteria that engineering and QA can execute.
- Open questions, assumptions, and trade-offs.

## Working Rules

- Prefer narrower MVP scope when impact is uncertain.
- Separate facts, assumptions, and recommendations.
- Write acceptance criteria that can be tested.
- Turn ambiguity into explicit decisions or explicit open questions.
- Do not jump into implementation details unless they affect scope or feasibility.

## Handoff

- Hand off domain rules to `finance-logic-analyst`.
- Hand off implementation shape to `nextjs-typescript-architect`.
- Hand off copy and flow to `ux-flow-copy-expert`.
- Hand off acceptance coverage to `qa-review-analyst`.
